Introduction to Joins:

In the lectures so far, we have been writing queries that retrieve data from a single table only.
For this lecture, let us consider an Example:

There is a Students table:

There is a batches table:

We are required to get the information of all students belonging to a batch named B2.

Approach 1:

  1. Get Bid using batch name
Declare @Bid INT;
select @Bid =id
from batches
where name ="B2";
  1. Get all students belonging to relevant Bid.
select * 
from students
where bid=@Bid;

So, this approach will give us the information.

Problem: It is making two calls to the server.

Generally, In real systems, a user is there, an application server and database is there. When a user make a request to the server, the server internally make call to the database server.

But network calls are going to be slower than CPU operations.
Therefore not a good solution.

Approach 2:
Instead of having two seperate tables, Let’s have a single table for students and batches information.

Problem:

  • Redundancy.
    • Slow Performance
    • Higher chance of inconsistency of data.
  • Bad database design.

Approach 3:(Ideal Solution - Joins)
Join helps in merging the two tables virtually for the purpose of query and not actually creating a new table in the database.

Let’s take students and batches table.
So Join helps in linking the two tables on appropriate columns(here bid is making foreign key to the batches table).
Using this information our query will help us make a virtual table.

To combine two rows, the first row of student will consult to all the rows in batches table and if any of the batch id matches, We will copy that corresponding row to the final table.
Repeat the process for all the rows in students table.

If any of the row in students table didn’t match with the batches table, then it will not come in the final table.

orders.customer_id= customers. id

Self Join:

Self join is a table making an Inner Join with itself.

Let’s assume there is a students table where every student has a peer reviewer.

We are required to get name, reviewer_name for all students.

Now, Let’s hypothetically assume peer_review is a seperate table,

Now we can simply do a inner join between these 2 tables and get the answer.

select s.name,r.rname
from Students s
join reviewer r
on s.r_id=p.id;

But the only difference between real and this hypothetical scenerio is that the peer_review is not a seperate table.
Therefore we can join students with students table and give an alias.

select s.name,r.name
from students s
join students r
on s.r_id=r.id;

In a self join we can’t avoid creating aliases because we have to join a table with itself only.

Let us consider another example from our shared sample databases in the mysql workbench:

There is an employees table

Here reports_to column in employees table is the employee_id of their manager.

We are required to get the name of every employee with the name of their manager.

We can simply use Self Join here.

Select e.first_name,m.first_name
from employees e
join employees m
on e.reports_to=m.employee_id;

Compound Join:

Compound Joins have multiple conditions in the ON clause.

select * 
from table A
join table B
on {condition1}
and {condition2}

Let’s take an Example:
There are two tables, T1 and T2.

Select *
from T1
join T2
on T1.a=T2.aa
and T1.b=T2.bb;

Syntactic Sugars:

USING keyword:

There is a students table and a batches table:

select *
from students s
Join batches b
on s.bid=b.bid;

This query can be re-written with the “using” keyword as follows:

USING keyword:

select *
from students 
Join batches 
USING(bid);

When both the tables have the same column which has to be joined, then instead of ON clause we can also apply USING clause. It reduces the amount of lines in our code.
We can also put 2 columns in USING clause.

Natural Join:

No need to write any condition. No ON and USING.
Automatically join based on columns with same name.

select *
from students 
NATURAL JOIN batches;

It goes through both the tables, find the column with the same name on both the tables, tries to do the join based on that column.

  2. Right Outer Join

Cross Join:

(Cartesian Product)
Cross Join is every row put against every row(n*m).

If students has 9 rows and batches has 10 rows, The cross join would have (9*10)=90 rows.
It is needed when we want every row from both the tables.

select *
from students
Cross Join batches;

CROSS JOINS are slower than INNER JOINS because they are practically going from every row of A to every row of B.
