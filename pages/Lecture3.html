SQL Data Types:

1. Integers:
* There are different types of Integer data type that exist based on their size.
* TINY INT: 1 Byte (-128 to 127), (0 to 255).
* (0 to 255) is called UNSIGNED TINY INT ( more like a char/byte)
* SMALL INT: 2 Bytes (-32k to 32k) (Doesn’t really exist)
* MEDIUM INT: 3 Bytes (-8M to 8M)
* INT: 4 Bytes (-2B to 2B) (normal INT)
* BIG INT: 8Bytes (-9Z to 9Z) (normal long long INT)

2. Floating Points:
The reason for using floating point numbers are:

  • Huge range of numbers they can store.

  • They store scientific notations.

  • Used in scenerioes where approximation is ok.

  • Sometimes we write extra logic to handle approximation.

    1. Decimal(p,s)
    * p: Total length/precision (1 to 64)
    * There are 65 digits, so maximum power of 10 is 64, Now we can say decimal store more than floats, since float has max of 1038
    * s: How many values after decimal that we want to store.
    * Example: 302.04, p: 5, s:2
    * Let’s say internally MYSQL stores 30204 then also MYSQL will be able to know exact decimal point.
    * It allows MYSQL to store exact values.

    2. Float:
    * It stores 4 Byte number.
    * (-3.4* 1038 to 3.4* 1034)

    3. Double:
    * It stores 8 Bytes.
    * (-1.7* 10308 to 1.7* 10308)

3. BOOLEAN: (BOOL or BOOLEAN)

  • Internally a boolean is nothing but a Tiny int.
  • Size of boolean: 1Byte
  • It can store 2 values: True or False/ 1 or 0

4. ENUMS:
* A set of constants.
* With enum compiler is going to ensure that we don’t end up inserting something that is not a valid value.
* Value should be amongst the given set of values.
* For Example: In a cars table, there is a car_type column and we specify them using enum. If enum is defined as enum(“BMW”, “sedan”, “audi”) and if we try to put any value other than this, it will throw error.

5. Date and Time:
Date:
It only allow to store a date, 28/3/2022
Time:
It Only allow to store the time part, 6:00pm
DateTime:
It allows to store both date and time, 28/3/2022 6:00pm

6. TIMESTAMP:

  • In MYSQL, Timestamp is of 4 bytes.
  • Because the number of seconds since 1970> range of timestamp, all of the databases that are using timestamp as the column to store a particular time are going to overflow after 2038.
  • https://en.wikipedia.org/wiki/Year_2038_problem(Read more from this).

7. BLOBS:

  • Binary Large Objects.
  • It Stors files in database.
    TINY BLOB: 255B
    NORMAL BLOB: 65KB
    MEDIUM BLOB: 16MB
    LONG BLOB: 4GB
  • Often a file store is going to be a better choice.
  • Store metadata of file(fileURL, fileId) in database and store file in a file store(AWS S3)

Normalization:

  • Process to organize the data in a a database.
  • A techinque to reduce redundancy of data in a database.
  • Helps to create better database schema(Schema with reduced redundancy).

Anomalies with Redundancy:

Let’s understand with the help of scaler example:
We have a table called students.

Here Instead of creating seperate tables for students and batches, We merged both tables in one, students table have information about batches as well as students.

Here everytime batch_name and batch_instructor is repeated for all the studens.

The problem is redundancy, the information about batch_name and batch_inst is redundant.

Different problems due to Redundancy:

1. Insertion Anomaly:
We can’t create a batch till no student is there.
We have to put NULL in all the columns of students information because of that id becomes NULL and Primary key can’t be NULL.

2. Deletion Anomaly:
Let’s consider a use case where,
Scaler decided that all the students of batch A and C should go to batch D with instructor B and id 3.
We have to make the changes in batch_id, batch_name and batch_instr column of all the students.

Now, the problem is there is not a single information left about A, C anymore.
Moving the students, deleted the batches, which is unintended side-effect.

3. Update Anomaly:
Now let’s say Scaler wants to change the instructor of batch A to G.

Redundancy increases the probability of data inconsistency.

Normal Forms:

1NF:

  • Every column should have atomic value.
  • No multi values attributes.

Let’s consider a students table:

Solution1:
Create multiple columns each for storing seperate phone_numbers.

But We are having too many NULL, sparse table.

Solution2:
Create seperate table for storing phone_numbers.

If we have the id column, the primary key of phone-number table is Id.
If we don’t have this, then st_id and ph_no combined will be the primary key.

Benefit of this: It is sorted on st_id, that means if we want all the ph_no of a particular student, it’s going to be faster.

2NF:

Let’s take a scenerio.
Scaler is storing a database table called mentor_sessions.

Primary key: First 3 columns because 1 student with 1 mentor can have mutiple sessions.

Now, we can say the attribute mentor_name can just be found via mentor_id. This mentor_name is actually not depending on all the 3 columns but just mentor_id, a part of primary key.

According to 2NF, No non-prime attribute(attribute not part of primary key) should depend on subset of primary key.
All the non prime attributes should depend on complete primary key.

Here, mentor_name is dependent upon a subset of primary key. This is not allowed.

3NF:

According to 3NF, No non prime attribute should indirectly depend on primary key.

Let’s say we have students table,

Primary key: id.
batch_id-> batch_name
id->batch_id

Here, batch_name depends on batch_id which depends on id i.e. primary key.
So the table is not in 3NF.

Insert:

SQL is case insensitive.
INSERT=insert=INseRt 

insert into [table_name] values 
(           )

Let’s consider an example from our shared sample databases in the mysql workbench:

We are required to insert a new customer.

insert into sql_store.customers 
values (11, "naman", "bhalla", '1998-01-02, 123', 'ABC', 'XYZ', 'HR',1134),
(12, "naman", "bhalla", '1998-01-02, 123', 'ABC', 'XYZ', 'HR',1134);

Run this (command+enter) or run query button.
There are 2 new customers inserted in database.

In table defination,
Id is auto_increment and primary key, auto_increment is we can generate the primary key ourself whatever is the past key that we had, just add 1 to it and that is going to be the new primary key.
Therefore database will automatically generate the value. 

insert into sql_store.customers 
values (default, naman, bhalla, '1998-01-02, 123', 'ABC', 'XYZ', 'HR',1134),
(default, "naman", "bhalla", '1998-01-02, 123', 'ABC', 'XYZ', 'HR',1134);

Another variant:
It is making the query more readable. 

insert into sql_store.customers (customer_id, first_name,last_name,birth_date,phone,address,city,state,points)
values(default, "naman", "bhalla", '1998-01-02, 123', 'ABC', 'XYZ', 'HR',1134),
(default, naman, bhalla, '1998-01-02, 123', 'ABC', 'XYZ', 'HR',1134);

Now, there are some columns that are nullable also, (birth_date, phone etc), we can remove that. 

insert into sql_store.customers (first_name,last_name,address,city,state,points)
values("naman", "bhalla", 'ABC', 'XYZ', 'HR',1134),
(naman, bhalla, 'ABC', 'XYZ', 'HR',1134);

It will put the values, only difference is the values which we haven’t provide have NULL.

Another benefit of giving the column names is:
These columns can be in any order.
This means that if we specify the column name ourself, the column names need not to be in the same order as in table. 

insert into sql_store.customers (first_name,last_name,address,city,state,birth_date,points)
values("naman", "bhalla", 'ABC', 'XYZ', 'HR','1998-01-02',1134),
(naman, bhalla, 'ABC', 'XYZ', 'HR','1998-01-02',1134);

Update:

We want to change the value of something that is stored.

Update [table_name] set (give values) condition(optional)

Example:
update students set name='naman'
No condition: Therefore names of all the students become naman.

Now, We want to update the name of customer_id 21 to manish yadav.

Update customers set first_name='manish', last_name='yadav' where customer_id=21;
This updates the name of the customer with id=21.
