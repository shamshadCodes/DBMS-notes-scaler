Introduction to Database Management System.

Let’s understand the word DBMS, Database Management System.

Now, Management and System are already defined words.
A management system is going to help us manage something.

Let’s understand word Database now,
Database is composed of a word Data.

Data is any information about something.

Let’s take an Example: A user clicks on a particular button in a website now the information about that user clicking on a button is Data.

Therefore, Data is any information that is important for someone to do their work.

Another Example can be, For Scaler the number of students attending a class is a data.

Likewise, Facebook also tracks a lot of data for us. As soon as a user scroll, likes a picture, upload something even that is the data for facebook.

We can say that, Every information that is around the world howsoever small it may be is a data.

Now, Let’s imagine the Scaler software system,

So any information that is helpful for Scaler to make it’s business decisions or to execute it’s business decisions is a data for scaler.

We all know about Airbase: It’s a Place to store air craft.
Similarly, Database: It’s a Place to store Data.

A database is a collection of related data.
Let’s again imagine the Scaler codebase, all the different related data items for Scaler might be,Students, mentor, assignment,classes, questions.
All these could be put in one database.

But, Scaler will not be storing the analytics related data, like how many people clicked on a particular button, scroll of a user etc in the same database because it’s not the related data.

Hence, Databases are seperated by the purpose they serve.

DBMS is the software system that is used to manage database systems.
Hierarchy of the information:


DBMS like MYSQL, PostgreSQL,MongoDb all of this allow to have multiple databases inside them. In every database we can have multiple data that are related to each other.

Types of DBMS:

The purpose of a DBMS is to store databases and provide ways to interact with data.

An OS can be used like a DBMS.
In one DBMS there can be different databases, likewise every database is a different folder.
Now In folder1, User create one file called students.txt, then there is Instructors.txt file, then assignments.txt. So each of these is a text file.
Like in students.txt file, every row of students.txt have information about 1 student similarly instructors and assignment.txt also.

The real databases or the very first database that came into existence is: file based DBMS.

1. File Based DBMS:

  • It stores data as files on disk.
    Let’s take previous example, the information about students might be maintained in 1 file, similarly information about instructors can be maintained in one file.

Cons of file based system:

  • Retrieval is not Easy:
    In file based DBMS, traversal happens one by one.
    Since file is nothing but a series of bytes so we can start reading from the very first byte till the place where the result is.
    Because the traversal is sequential hence retrieval is slow and inefficent.

  • Possiblity of error:
    Let’s take students and assignement file.Now, we need to find a relation that which student has which assignment, we have to store a relation between students and assignments.
    Here Let’s say this is the students file.

    In this file, there is a student with id 1, Name ‘Naman’, email ‘xyz’ then assignments are there.

Now, Let’s imagine there is no assignment with no. 241 in the assignment file.
But for that the file system is not going to through any error because File system is just storing bytes that’s it.

Therefore there is a possibility of errors specially errors while storing relations. Those realtions might not have the data on the other side. This is actually Inconsistency.

Let’s say student data is needed by finance team. So finance team also has students data. Now the students data is also needed by instructors team, so instructor team will have their own folder and they will also have students data.
This is also needed by business team. Similarly business team also has students data in their own folder.

Now, The problem is Duplication. And because of Duplication, Maintaining consistency becomes a problem.

  • Inconsistency and Duplication:
    Let’s say one student, Naman decided to change his email.
    Fot that scaler made an update on the students table as well as on the Finance table but was not aware of the instructor’s table. So now at two places Naman has different email id’s and there is no way to know which one is correct or the latest one. This can lead to inconsistent issues.

The same information has two different values at two different places and it’s difficult to examine which one is actually correct.

  • Security:
    There is no such security is offered by file sytem since anyone who opens the file can read the whole information, they have the control of whole database then.

  • No support for Concurrency:
    In a file based DBMS, if two people trying to access the data at the same time, they cannot able to do that.
    Since the data eventually is stored in DISC.

Let’s understand How a Disc works:
In disc there is a pointer and this pointer is basically moving, it traverses over the complete disc.
Pointer goes to the place where the user wants to find the data, it gives the data and then moves ahead. And there is only 1 pointer per disc.

Disc can access only one information at one time, because pointer will only go to one particular place at one time. So, there is no possibility of concurrent access.
The pointer cannot go to more than one place at the same time, it will first serve one request then another request.
That means in real world systems, using file system as a DBMS is not a good idea.

So that is how came the use case of Dedicated DBMS, rather than relying on file systems to store our data these days, more sophisticated, complicated and feature full DBMS came into existence.

2. Dedicated DBMS:

Features:
1. Backups: Data Backups are very important thing because In reality Data loss is very common, the hardware in which the data is being storted can get messed up, the disc on which data is stored might get corrupted.

2. Concurrency: These days concurrency is very important because the scale of applications has become huge.

3. Security: Support for Encryption, even file level encryption etc are properties that are provided by DBMS these days.

4. Efficient Storage and Retrieval: Different databases like MongoDb, MYSQL, PostgreSQL have different efficency.
For Example: MongoDB’s are most efficient for some type of operations, similarly MYSQL in another set of operations.

Now let’s understand,
How does the database store data. Does data never go to Disks in that case:
DBMS is a software system, it’s an application, so it has a copy or it has some part of the data in memory. Every operation doesn’t involve directly going into the disc, it has memory in front of it to speed up the operation, that’s how real DBMS work these days.

Non-Relational DBMS(NO SQL DBMS):
Relational Databases don’t follow the relational model.

In relational database we have a table, if we want to find something in table, then searching starts in linear fashion, row by row and then get the data. That’s not efficient in all the cases.

Therefore there are many DBMS that exist these days that don’t actually store data in rows, store data in a different way, which is suitable for different type of operations. They are known as NOSQL DBMS.

Based on how they stored data, they are characterized further.
Types of NOSQL.

  1. Graph Based: Neo4J is the example.
  2. Columnar: Where the data is not actually stored in rows, but in columns.
    Example: Cassandra
  3. Document database
    Example, mongoDb
  4. Key value database.
    Example: Redis

Always start with SQL Database first and then contradict why SQL will not work.

Relational Databases:

  • This is the most widely used model.
  • The model behind SQL.
  • SQL is the language for Relational databases.
  • Represent data as collection of relations.

Firsty database is the collection of data whereas relational databases are collection of relations.
This implies that relations is nothing but data.

According to relational database, a Relation is nothing but table of values.

Every table we see in MYSQL database, that is a relation.

Now, each of these rows is actually an information of an individual student or an individual batch where each of these is known as an Entity.
Each row is an information about a real Entity.

Relational databases say’s that, collect data into relations. A relation in logical form is nothing but a table, and inside the table there are multiple rows.

Terms of Relation:

1. Relation: Relation is nothing but Table.

2. Attributes: Properties of relation: Columns of the relation.

3. Tuple: Rows of the table.

4. Degree: Number of attributes of a relation: Number of Columns of a table.
To find the degree of a relation, Just count the number of columns.

5. Cardinality: Number of tuples in the relation or the Number of rows in a table.

The cardinality of a table change more often than degree.
Because whenever we insert a new value, the cardinality is going to change.

Properties of a Relation/Table:

1. Each row is unique: It means atleast one value should be different. We cannot have a relation called students where every information is same.

There are 2 students whose psp is 90, this leads to duplication.

This is resolved by having id column.

2. The values in cells should be atomic: No list/JSON/collection.

Phone numbers are not single value, they are list of values.
Traditional SQL databases, not allowed to have this information.

MYSQL 8 still don’t have array, but allow JSON.

Prefered way:

3. Order of columns shouldn’t matter: Order of the column should never be assumed or should never be relied upon.
One should name the particular column which is to be accesses since no one should expect a relational database the same order of columns.

The order might differ.

Theoretically, Should never hardcode the order of columns.

4. Order of rows should not matter: We should never expect that if we make a query, then query gives output in the exact same order the next time we run the query.

These constraints exist in relational model because relational mode of DBMS is based on a mathematical concept: Set Theory.

Relation=Set
Every table/relation is a set.
In Set order of values doesn’t matter
Set, {A,B}={B,A}
List, [A,B]!=[B,A]

If every set is a relation that means every tuple is a value inside set and every table is nothing but a set of tuples.
Set,{(T1),(T2),(T3)}
Order of these tuples will not matter. That’s why order of rows should not matter since every row is nothing but a tuple.

Cardinality of Set: Number of values in Set,
Cardinality of relation: Number of rows in relation.

The set{A,A,B,C}={C,B,A}

In HashSet we cannot put multiple keys with same values because duplicate don’t matter.
Same in relation, each row is 100% unique.

In Students table, multiple students are there.

What are the columns in this table that can uniquely identify a column.
Email,Phone_number -> Key

Keys:

Attribute/set of attributes that can uniquely identify a row in a relation.

In this table,

Types of Keys:
1. Super Key: A super key most minimal constraints.
It is a set of attributes that can uniquely identify a relation.

2. Candidate key: A key of the minimum size that can uniquely identify a relation.
Minimum size: If we remove any of the attributes, the resulting attribute will not be a key.

Since after removing 1 from (email,phone_no) still we are still able to identify the row, therefore (email, phone_no) is not a candidate key, but it is still a super key of minimum size.
Candidate key is super key of minimum number of attributes

3. Primary Key: One of the candidate keys chosen by the database architect/creator as the key of the table.
Only/Exactly 1 per table.
We have 2 candidate keys, the database administrator will choose 1 from them,

But email is also not a good option, since email is a string and string is actually repeated multiple times in the mapping tables which is wastage of storage.

Now, also a user can change email, therefore email is not a good choice from design perspective.

Ideally primary key should never change.

In this case, No column is there which will not change.
In practicality, we create another column called student_id.

Now, even if the email of a person change student_id, remain same.

Student_id is super key, candidate key and choosen to be Primary key
Now, if we take student_id as primary key we are just repeating integer not string.

Primary keys are actually chosen to be automatically generated integers or some value which is not the user’s information.

4. Composite Key: A key with greater than or equal to 2 columns.
In above example, primary key was not a composite key.

5. Foreign key:
Now, Let’s imagine students table, and every student has a batch in which they are part of, every student can be part of 1 batch only.

Primary key of students table:
id

Now, the responsibility of batch_id is it’s going to uniquely identify batches. But batches are not stored in students table instead in some other table.
Therefore, this batch_id which is uniquely identifying batches(stored in different table) is known as foreign key.
Foreign key is not associated to one’s table, it is associated to other table.

Primary key of Batches table:
id

SQL:

Structured Query Language.

Query Language means to query over something.
It’s a language that is used to query over relational databases since relational databases are in structured format as in rows and columns therefore It’s called Structured Query Language.

Language used to manipulate data(insert,update,read,delete).

Create Database:

Create database scaler_class;
Every SQL statement will end with semicolon.
Execute any SQL statement using command+enter.

Just refresh, and can see a scaler_class.

Now, the problem is if someone shares this particular file with anyone and the person runs it, They will see an error.
Because it’s already created.

So don’t just write create database but instead,
create database if not exist scaler_class;
Now it has not shown error, just warning.

Delete Database:
Drop database scaler_class;

Same problem again therefore,
Drop database if exists scaler_class.
Now, just showing warning, not error.

Use the following command to select a database.

Use scaler_class;
Now, Whatever queries we are going to write they are going to automatically assume that use scaler_class database only.

For Example: Create a table.

create table students(
student_id INT(data type),
first_name VARCHAR(10),
last_name VARCHAR(10)
);

Ideally, whenever we create a table, create it with Primary Key only.

create table students(
student_id INT(data type) PRIMARY KEY,
first_name VARCHAR(10),
last_name VARCHAR(10)
);

Now, it created this table in scaler_class database because we have used scaler_class.

Now, Let’s say we want to create this table in bakery database.

create table bakery.students(
student_id INT(data type) PRIMARY KEY,
first_name VARCHAR(10),
last_name VARCHAR(10)
);

Whenever we use the table, prefix it with name of database or use USE keyword.

DROP table scaler_class.students;

Also, write if not exists and if exists:

create table if not exists scaler_class.students(
student_id INT(data type) PRIMARY KEY,
first_name VARCHAR(10),
last_name VARCHAR(10)
);

DROP table if exists scaler_class.students;

Now, create one more table batches,

CREATE table if not exists batches(
batch_id INT PRIMARY KEY,
batch_name VARCHAR(20),
instructor_name VARCHAR(20)
);

We have 2 tables in scaler_class database.

Now, we want that every student should have batch_id.
We can drop the table and create again with batch_id column.

create table if not exists scaler_class.students(
student_id INT(data type) PRIMARY KEY,
first_name VARCHAR(10),
last_name VARCHAR(10),
batch_id INT
);

Problem: Existing data will be gone, We can’t always do that.

To avoid data loss, alter the table after creating in case of any change.
Modify table: ALTER keyword is used.

alter table students
add batch_id INT;

Run this and it’s updated.

batch_id is foreign key.

To add foreign key, use the following statement

Alter table students
add foreign key fk_students_batches (batch_id)
references batches (batch_id);
  • Foreign key naming convention:
    FK_Table1Name_Table2Name (column).
